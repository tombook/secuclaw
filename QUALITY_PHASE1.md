Phase1 Quality Assurance Standards and CI/CD Configuration

本文件给出 Phase1 阶段的完整质量保障规范与持续集成/持续交付（CI/CD）配置，便于直接将配置复制到项目根目录使用。内容覆盖：代码评审要点、单元测试覆盖与用例规范、GitHub Actions CI、预提交钩子，以及发布门禁标准。若项目已有现成实现，请以本规范为准进行差异对比与落地实现。

1) 代码评审标准 checklist
- 需求对齐：改动是否明确对应需求/用户故事，变更范围是否合理，避免“返工式”改动。
- 风格与可读性：遵循项目现有编码风格、命名规范、注释充足且必要；避免过度耦合与复杂度过高。
- 设计与模块性：职责分离、接口设计清晰、低耦合高内聚，避免全局静态状态滥用。
- 容错与边界：输入校验、错误处理、边界条件覆盖、异常路径考虑到；考虑并发/异步场景的鲁棒性。
- 性能与资源：关键路径的时间复杂度/空间复杂度评估，避免在热路径中引入不必要的性能开销。
- 安全性：参数/输入的安全校验、避免信息泄露、XSS/SQL注入等常见风险点；依赖安全扫描结果纳入评审。
- 测试覆盖与可测试性：评审测试用例覆盖范围、可重复执行性、测试依赖最小化。
- 变更可追踪性：Commit 信息清晰，PR 描述准确，更新变更日志/版本记录。
- 依赖与版本管理：锁文件变更合理；对重大版本/潜在破坏性变更进行风险提示。
- 文档更新：API/使用文档、部署文档、架构图等必要文档同步。
- 构建与部署相关性：变更能在 CI/CD 流程中正确构建、打包、部署，避免“上游通过”，下游失败的情况。

2) 单元测试覆盖要求、测试用例规范
- 覆盖目标：核心业务逻辑的单元测试覆盖率不得低于 80%，关键路径和高风险模块不少于 90%；全局覆盖率以报告为准。
- 覆盖粒度：按模块/功能拆分测试用例，单元测试应覆盖边界条件、异常路径、正常流程、并发场景等。
- 测试框架与断言库：优先使用项目现有测试框架（如 Jest、Vitest、Mocha），并统一断言风格。
- 测试结构与命名：tests 目录结构统一，测试文件命名应为 xxx.test.ts/xxx.spec.ts；测试用例名称应描述性强。
- 外部依赖与副作用：对网络、数据库、文件系统等外部依赖使用 Mock/Stub；避免对真实系统的写操作。
- 数据与 Fixture：测试数据应可重复、可解释；必要时提供固定 fixture 文件。
- 运行方式：通过 npm test/yarn test 触发，确保在 CI 中同样覆盖。
- 失败处理：测试失败应明确回传失败用例、期望结果与实际结果。
- 覆盖报告：输出覆盖率报告（如 coverage/coverage-summary.json），并将摘要写入 CI 日志或报告中。
- 测试用例模板（示例）
  - Title: 验证 sum 函数对常量输入的和正确性
  - Purpose: 确保 sum(-1, 1) 返回 0，且边界条件正确处理
  - Preconditions: 无
  - Steps: 调用 sum(-1, 1) 并断言结果为 0
  - Expected Result: 返回 0
  - Cleanup: 无
  - Tags: unit, core

3) GitHub Actions CI 配置（lint、type-check、test、build）
- 目标：在 PR 和推送时自动执行静态分析、类型检查、单元测试与打包构建。
- 集成要点：采用 Node.js 环境，支持多版本 Node 的矩阵测试，缓存依赖以提升速度。
- 典型前提：项目根目录应提供 npm 脚本 lint、typecheck、test、build。
- 配置文件将直接放在 .github/workflows/phase1-ci.yml，内容可直接复制使用。

4) 预提交钩子配置
- 使用 pre-commit 框架在本地提交前执行校验：eslint、prettier、json/yaml 校验等。
- 需要在本地环境安装 pre-commit：pip install pre-commit; pre-commit install。
- 变更后每次提交前会自动执行 lint/格式化/校验等任务，确保提交质量。

5) 发布门禁标准
- PR 审核：至少两位以上开发者参与代码评审，且通过 CODEOWNERS/批准流程。
- CI 通过：所有 CI 子任务（lint、type-check、test、build）必须通过且没有警告为止。
- 安全与依赖：npm audit 或等效安全扫描通过，敏感依赖变更需额外评审。
- 发布条件： Tag 版本发布时，符合语义版本控制（SemVer），且变更日志（CHANGELOG）更新完成。
- 日志与变更：发布前需生成发行说明，更新文档与部署脚本，确保回滚可控。

使用说明
- 将 QUALITY_PHASE1.md、.github/workflows/phase1-ci.yml、.pre-commit-config.yaml 直接复制到项目根目录相应位置即可使用。
- 如需自定义 Node.js 版本、测试命令或打包流程，请在相应文件中修改，确保与项目实际脚本命令一致。
